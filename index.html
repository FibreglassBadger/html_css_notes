    Images and Links

    Anchor elements are used to create links < a >for example< /a >

    It needs a HTML attribute which is usually made up of two parts, a name and a value. This example uses a href hyperlink reference. The value of the href attribute is the destination the link goes.

    For example < a href="https://www.theodinproject.com/about">click me< /a >

    Anchor tags can be used to link all kinds of resources from the web ie images videos etc

    The target attribute tells the link where to open. By default it will open the link in the current tab. For a new tab you can set it to _blank.

    For example < a href="https://www.theodinproject.com/about" target="_blank" rel="noopener noreferrer">click me< /a >

    click me

    Rel is used to describe the relationship between the page and the linked document.

    noopener prevents access for opened link to previous page

    noreferer does the same vice verser. Done as good practise even though modern securty doesnt need it.

    Absolute links are pages to another site and will always be made up of protocol://domain/path eg https://www.theodinproject.com/about

    Relative links are ways to navigate your own website. Can be shorter eg pages/about.html

    Images

    To display an image in HTML we use < img > which is self closing. It uses src attribute similar to hrf. It can be embedded in absolute and relative paths.

    eg < img src="https://www.theodinproject.com/mstile-310x310.png" >

    alt= (Alternate) atribute must be used after the image location to decribe what the image shows, helping the visually impaired.

    eg < img src="https://www.theodinproject.com/mstile-310x310.png" alt="The Odin Project Logo" >

    Parent directories- To use an image that is not in the directory of the target web page then the following must be used < img src="../images/dog.jpg"> or ../targetfile so that it searches for it from the parent directory

    Commit Messages

    Commit messages will be viewed by other developers working on your code to see what changes were made and why. They are good for reminding you of the changes you made and why

    A bad commit would be 'fix a bug' a good commit has a subject 'this is a change I made to the codebase' and body describing what your commit fixed

    The messages need to be clear and concise so that they are easy to understand.

    Commits are like snapshot of your code and its action. This should show a time line of your progression. Commit when you get a piece of code to function, fix a typo or fix a bug. There will be times when your code may completely break but going back through your commits (timeline) to before the code broke would be very helpful

    The seven rules of commits are:

    1. Separate subject from body with a blank line
    2. Limit the subject line to 50 characters
    3. Capitalize the subject line
    4. Do not end the subject line with a period
    5. Use the imperative mood in the subject line
    6. Wrap the body at 72 characters
    7. Use the body to explain what and why vs. how
